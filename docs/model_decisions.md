# Reduced topology model decisions

This document records the counting model implemented by the reduced-topology
pipeline, which is the current and only implementation. It also explains the
historical legacy multiset-bundle model that the reduced model replaced.

The purpose is to count useful two-terminal RLC one-port topology classes, while
stripping away local redundancies that are obvious for ideal primitive R, L, and
C elements.

## Historical: legacy model (removed)

The legacy counter, now removed in full (`docs/plan/02-cleanup/02-legacy.md`),
counted connected two-terminal support graphs and assigned non-empty
component-count bundles to support edges. In `lc` mode a bundle was a triple
`(r, l, c)` with arbitrary non-negative counts subject to the global
component budget.

That meant the legacy counter treated these as distinct bundle labels:

```text
R
R || R
R || R || R
```

and similarly for `L` and `C`. This behaviour is documented here only as
historical background explaining why the reduced model differs; it is not
retained or reproducible by the current source.

## Reduced model

The reduced model counts two-terminal RLC one-port topologies after local
primitive series and parallel redundancies have been removed.

A raw network has:

- two external terminals;
- zero or more internal nodes;
- undirected primitive branches labelled `R`, `L`, or `C`;
- no self-loops;
- no component values.

The two terminals form an unordered pair. Swapping the terminal names does not
create a new topology. Internal node names are arbitrary and do not create new
topologies.

## Terminal relevance

A terminal-labelled support graph is terminal-relevant iff every support edge
lies on at least one simple path between the two terminals.

If a terminal-labelled support graph has a dangling branch, dangling tree, or
pendant blob, reject the whole terminal-labelled graph.

Do **not** prune the irrelevant part and keep the remainder. Pruning would turn
one generated object into another generated object and would create duplicate
representations. The relevance test is an accept/reject gate.

## Simple primitive bundles

A reduced primitive bundle is a non-empty subset of `{R, L, C}` placed in direct
parallel between the same two support nodes.

The only primitive bundle labels generated by the reduced model are:

```text
R
L
C
R || L
R || C
L || C
R || L || C
```

Thus `r`, `l`, and `c` each take values `0` or `1`, and not all are zero.

The reduced model does not generate:

```text
R || R
L || L
C || C
R || R || L
R || C || C
```

because repeated same-type ideal primitives in direct parallel reduce to a
single primitive of that type, with a different component value.

Mixed primitive bundles remain distinct:

```text
R || L
R || C
L || C
R || L || C
```

`R || L` is not the same topology as `R -- L`.

## Series spans

The series-side analogue of a primitive parallel bundle is a local **series
span**.

A series span is a maximal local chain of two-terminal factors separated by
internal nodes of incidence 2 in the current reduced graph. It is local: the
span need only lie on at least one simple path between the external terminals.
It does not need to lie on every terminal-to-terminal path.

This distinction matters. In:

```text
      R---L
s ------------- t
 \             /
  ----- C -----
```

the upper `R--L` arm is a series span even though the lower `C` arm gives another
simple path from `s` to `t`.

Do not use only terminal-separating articulation nodes to find series spans; that
would miss spans inside parallel arms.

## Series commutation and primitive duplicate merging

A series span is treated as an unordered series composition of its factors.
Therefore:

```text
R -- L
L -- R
```

have the same reduced signature.

Likewise, all permutations of a primitive series chain have the same reduced
signature:

```text
R -- L -- C
C -- R -- L
L -- C -- R
```

Within a series span, repeated primitive singleton factors merge:

```text
R -- R       -> R
L -- L       -> L
C -- C       -> C
R -- L -- R  -> R -- L
```

More generally:

```text
R -- A -- R  -> A -- R
```

where `A` is any other series factor, because the two primitive `R` singleton
factors can be combined.

## Duplicate compound subnetworks do not merge

Do not merge repeated compound subnetworks merely because they are repeated.

In parallel:

```text
(R--L) || (R--L)
```

is not reduced to:

```text
R--L
```

In series:

```text
(R||L) -- (R||L)
```

is not reduced to:

```text
R||L
```

The rule is:

```text
collapse repeated primitive singleton R/L/C factors;
do not collapse repeated compound factors.
```

## Symmetry between series and parallel

The intended symmetry is:

| Composition | Operands commute? | Duplicate primitive singleton factors merge? | Duplicate compound factors merge? |
|---|---:|---:|---:|
| Series | Yes | Yes | No |
| Parallel | Yes | Yes | No |

Parallel operand order is already absent in the graph: arms between the same two
boundary nodes are unordered. Series operand order must be removed explicitly in
the reduced signature.

## What remains distinct

The reduced model should still count these as distinct:

| Case | Treatment |
|---|---|
| `R || L` versus `R -- L` | distinct |
| `R || C` versus `R -- C` | distinct |
| `L || C` versus `L -- C` | distinct |
| `(R--L) || C` versus `R || L || C` | distinct |
| `(R--L) || C` versus `R--L--C` | distinct |
| bridge/core networks versus series-parallel networks | distinct unless graph-isomorphic under the reduced signature |
| different L/C choices | distinct in the main `lc` model |

Component values are not represented. Different values of the same topology are
not counted separately.

## What is outside scope

The reduced model is not intended to solve general electrical equivalence.
Do not collapse networks using:

- Y-Delta or Delta-Y transformations;
- bridge-balance special cases;
- duality;
- Foster/Cauer transformations;
- algebraic equality of rational driving-point impedances;
- value-dependent cancellations or degeneracies.

Those may be studied later, but they are not part of the present reduced
counting contract.

## Implementation stages

1. Support-graph census and terminal-relevance filtering: implemented as
   `rice supports` / `support_census`. See
   `docs/support_graph_enumeration.md`.
2. Simple primitive bundle assignment: implemented as `rice bundles` /
   `simple_bundle_assignment_census`.
3. Canonical reduced signatures with local series and parallel reductions for
   individual assigned networks: implemented as `canonical_reduced_signature`.
4. Phase-3 orbit representatives merged by reduced signature into final
   reduced-topology counts: implemented as `rice reduced` /
   `reduced_topology_census`, currently computed and committed for the small
   golden slice `R <= 2`, `L+C <= 3`. Running the full `R <= 3`, `L+C <= 5`
   scope through this same enumeration is remaining work
   (`docs/plan/05-slices/04-r3-x5.md`), not an unimplemented stage.
